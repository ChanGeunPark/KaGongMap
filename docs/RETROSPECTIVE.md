# 회고 · 트러블슈팅 기록

카공맵을 만들며 가장 머리를 많이 굴린 문제 3가지를 기록합니다. 각 항목은 **상황 → 문제 → 시도 → 선택 → 결과 → 배운 점** 순으로 정리했습니다. README 의 "기술적 의사결정" 섹션이 *무엇을 골랐는지* 라면, 이 문서는 *왜 그 선택에 도달했는지* 입니다.

---

## 1. 비로그인 좋아요 — 카운트와 개인 기록 분리

### 상황

카공맵은 "보는 건 자유, 참여도 일부 자유" 원칙을 유지하면서, 좋아요는 **비로그인도 가능**하게 풀어둔 상태였다. 좋아요 수는 카페 핀 정렬의 기본 키이자 사회적 증명 역할을 한다.

### 문제

이 조합은 단순해 보이지만 두 가지가 정면 충돌한다.

- **익명 사용자도 카운트에 기여시키고 싶다** — 그래야 막 등록된 카페가 0 에 머무르지 않고 사회적 증명이 생긴다.
- **로그인 후엔 "내가 누른 카페" 가 마이페이지·필터에 정확히 떠야 한다** — 그러려면 누가 눌렀는지 식별 가능한 기록이 필요하다.

여기에 추가 제약이 더 있다.

- 같은 브라우저에서 좋아요를 *N* 번 눌러도 카운트는 1만 증가해야 한다 (중복 클릭 차단).
- 비로그인 상태에서 누른 좋아요를 로그인 시 머지하면 → **이중집계** 위험. 머지하지 않으면 → 익명 시절 기여한 카운트와 본인 기록 사이에 불일치.

### 시도한 옵션


| 옵션                                      | 문제                                                                     |
| --------------------------------------- | ---------------------------------------------------------------------- |
| (A) `cafe_likes` 한 테이블에 익명 user 까지 다 저장 | 익명 user 식별자(브라우저 fingerprint?)를 만들면 RLS 정책이 폭발. 익명 row 가 무한 누적될 위험도 큼. |
| (B) 비로그인 시 카운트 증가도 막기                   | 익명 트래픽이 데이터 누적에 0 기여 → 카페 핀이 새로 등록될 때마다 "좋아요 0" 으로 시작하는 문제 미해결.        |
| (C) 비로그인 → 로컬에만 저장, 로그인 시 서버로 머지        | 카운트는 이미 비로그인 시점에 RPC 로 ±1 됐기 때문에 머지하면 그 카페만 +1 이 두 번 반영 → 이중집계.        |
| **(D) 카운트와 개인 기록을 *물리적으로 분리* · 머지 안 함** | **선택.**                                                                |


### 선택한 설계

- **카운트** (`cafes.like_count`) — `bump_cafe_like_count(cafe_id, delta)` RPC 로만 변경. 누가 눌렀는지는 기록 안 함.
- **개인 기록** (`cafe_likes` 테이블) — 로그인 사용자만. `(user_id, cafe_id)` PK.
- **비로그인 source-of-truth** — `localStorage` (`[hooks/storage/likedStorage.ts](../hooks/storage/likedStorage.ts)`). 동일 브라우저 중복 클릭 차단은 여기서 처리.
- **머지 안 함** — 익명 좋아요는 식별 불가능하므로 머지 시 이중집계. 대신 카운트는 이미 반영돼 있으므로 *기여*는 살아있다.

추가로 UX 디테일:

- 낙관적 업데이트 후 **마커 캐시(`cafe_markers`)와 상세 캐시(`cafe_detail`)의 `like_count` 만 부분 패치** — 전체 refetch 회피. (`[hooks/useLikes.ts:54-71](../hooks/useLikes.ts#L54-L71)`)
- 실패 시 `onError` 에서 캐시 롤백 + localStorage 도 되돌림.

### 결과

- 비로그인 트래픽이 카페 카운트에 기여 → 새 카페가 발견되기까지의 콜드스타트 완화.
- 로그인 후 마이페이지 "내가 좋아요한 카페" 는 정확함.
- 이중집계 0건. 동일 브라우저 중복 클릭은 localStorage 가 막음.

### 배운 점

- **"한 테이블 = 한 가지 진실"** 을 깨야 풀리는 문제가 있다. 카운트와 개인 기록은 서로 다른 사용 사례 (정렬용 vs 마이페이지용) 라서 한 테이블에 묶을 이유가 없었다.
- 머지를 *안 하는* 결정도 설계다. "비로그인 → 로그인 머지" 가 항상 옳다는 가정을 깨야 했다.
- React Query 의 `setQueryData` 부분 패치는 강력하지만 **여러 캐시 키에 같은 필드가 흩어져 있을 때** 빠뜨리기 쉽다. 카드 리스트·마커·상세 모두 like_count 를 들고 있어서 한 곳만 패치하면 UI 가 갈라진다. 이런 경우 패치 함수를 한 군데로 모아두는 게 안전 (`patchLikeCount`).

## 2. AI 자동 제보 — 로컬 PC ↔ 클라우드 협업

### 상황

카페 등록을 사용자 제보에만 의존하면 초기 데이터가 깡통이다. 그래서 어드민이 **카카오 검색 → 카페 선택 → AI 가 영업시간 · 최소 주문 · 카공 태그 추정 → 검수 후 등록** 하는 자동 제보 흐름이 필요했다. 핵심은 "어드민이 점심시간에 클릭 몇 번으로 10개씩 채울 수 있어야 한다" 였다.

### 문제

AI 호출을 어디서 돌릴 것인가가 결정 안 됐다. 후보는 셋이었다.

- **(A) 클라우드 API 호출 (Vercel function 에서 `anthropic` SDK)** — 가장 정공법. 하지만:
  - LLM 비용이 카공맵 운영 비용으로 누적. 포트폴리오 단계에서 LLM 비용을 운영비로 떠안는 게 적절한가?
  - 함수 타임아웃 (Vercel hobby 10s, pro 60s) 안에 웹 검색 + 추론 끝나야 함. claude 의 deep research 류는 더 오래 걸린다.
  - 동시 요청 폭주 시 cost spike 방어 로직 추가 필요.
- **(B) 클라이언트에서 직접 LLM API** — API 키가 브라우저 번들에 노출. 절대 안 됨.
- **(C) 어드민 본인 PC 에서 도는 별도 프로세스 (로컬 브릿지) 가 LLM 호출** — 클라우드는 큐와 검수 UI 만 담당.

### 선택한 설계 — (C)

```
어드민 브라우저 (배포본 or localhost)
   │  ① 카카오 검색 → 카페 선택 → "AI 조사 시작"
   ▼
http://localhost:7332  ← 어드민 본인 PC 의 Hono 브릿지
   │  ② claude CLI spawn → 웹 검색 → JSON 응답
   │  ③ 어드민이 결과 검수 후 "대기 큐로 제출"
   ▼
POST /api/admin/auto-submissions  (어드민 가드)
   │
   ▼
cafe_submissions INSERT (status='pending')
   │
   ▼
/admin "대기 중" 탭에서 최종 승인 → 트리거가 cafes 생성
```

핵심 결정:

- **LLM 실행/비용은 본인 PC** — claude CLI 의 max 구독을 그대로 활용. 클라우드 운영비 0.
- **브릿지는 Hono + child_process.spawn** — 별도 프로세스라 타임아웃 자유. 1-2 분 걸리는 deep research 도 가능.
- `**BRIDGE_TOKEN` 으로 인증** — 브라우저가 localhost 를 부르더라도 토큰 일치하는 요청만 허용. 다른 사이트가 사용자 PC 의 localhost:7332 를 abuse 하는 걸 방어.
- **클라우드는 검수 → pending 큐만** — AI 가 만든 데이터를 *바로* DB 에 넣지 않는다. 어드민의 *한 번의 클릭* 이 무조건 끼게 설계. AI 가 헛소리해도 검수 단계에서 거른다.

### 시행착오

- 처음엔 클라우드 함수에서 fetch 로 anthropic API 호출 → 60s 타임아웃에 걸려 deep research 가 실패. (A) 의 한계 확인.
- claude CLI 가 stdin 으로 system prompt + 카페명 받고 stdout 으로 JSON 뱉도록 통일하기까지 프롬프트 튜닝이 오래 걸렸다. 마크다운 코드 펜스 안에 JSON 을 넣어주는 버릇이 있어서 파서가 깨졌고, "**JSON only, no markdown**" 을 명시하고도 깨질 때가 있어서 응답에서 `json ...`  펜스를 제거하는 sanitizer 를 한 단 더 둬야 했다.
- 모바일 미지원 — 브라우저가 본인 PC 의 localhost 를 호출해야 해서 폰에선 못 쓴다. 어드민 동선이 PC 중심이라 trade-off 수용.

### 결과

- 카공맵 운영비에 LLM 비용 절약. 어드민(=나) 의 claude max 구독으로 흡수.
- 카페 1건 조사 ~ 검수 ~ 제출까지 평균 **2-3 분** (대부분 검수 시간). 손으로 5-10 분 걸리던 작업 압축.
- 어드민 자리 비우면 동작 안 함 — 사용자 영향 0 (어드민 전용 기능).

### 배운 점

- **클라우드가 모든 걸 해야 한다는 가정을 버리자** — "내 PC 에서 도는 프로세스" 는 게으른 답처럼 보이지만, *내가 어드민이고 내 PC 가 항상 켜져 있다* 는 운영 사실을 활용하면 가장 합리적인 답이 되기도 한다.
- LLM 호출은 **검수 단계와 짝지어야** 한다. AI 결과를 즉시 production DB 에 넣는 자동화는 데이터 품질을 망친다.
- 외부 CLI 의 출력 sanitizing 은 항상 한 단 더 둘 것. "이번엔 잘 나오겠지" 가 깨진다.

> 더 자세한 셋업은 `[docs/AUTO_SUBMIT.md](AUTO_SUBMIT.md)`.

## 3. Tier 페이로드 분리 — 지도 초기 로딩 일정화

### 상황

카공맵의 메인 페이지는 지도 + 핀이다. 핀 클릭 시 카페 상세 모달이 뜬다. 단순한 패턴이지만 **카페가 많아질수록 첫 페인트가 느려지는** 문제가 있었다. 카페 한 row 가 영업시간 · 이미지 URL 배열 · 후기 카운트 · 카공 태그 · 좌표 · 주소 · 소개 등 굵직한 필드를 다 들고 있었기 때문이다.

### 문제

처음엔 `select * from cafes` 한 방으로 다 끌어왔다. 50개 카페 기준 페이로드는 측정해보면 **수백 KB**. 이건 두 가지를 동시에 망친다.

- **첫 페인트 LCP** — 지도가 뜨기 전에 카페 페이로드를 기다린다.
- **메모리** — React Query 캐시에 카페 50개의 전체 detail 이 떠 있다. 대부분 클릭도 안 할 카페인데.

스케일을 가정하면 더 심하다. 500개, 5000개 카페로 늘면 첫 페이로드가 선형으로 부푼다.

### 시도한 옵션

- **(A) 페이지네이션** — 지도 UX 와 안 맞는다. 화면에 보이는 핀은 viewport 기준이지 페이지 기준이 아니다.
- **(B) viewport bbox 쿼리** — 지도를 움직일 때마다 fetch. 미세한 panning 마다 요청이 터진다. 디바운스로 완화 가능하지만 캐시 무효화가 어려워진다.
- **(C) 클러스터링으로 한 번에 보이는 핀 수 줄이기** — 페이로드 자체는 안 줄어든다. 시각적 처리일 뿐.
- **(D) Tier 분리 — 마커는 lean payload 한 번에, 상세는 lazy fetch** — **선택.**

### 선택한 설계

Supabase 에 **두 개의 view** 를 만들었다.

- `**cafe_markers` view** — 마커가 필요한 7개 컬럼만: `id, name, address, lat, lng, like_count, min_order_amount, tags`.
- `**cafe_detail` view** — 모든 컬럼 + 이미지·후기 카운트 등 조인 결과.

호출 패턴:

- **Tier 1 — 첫 페인트** · `[fetchCafeMarkers](../lib/api/cafes.ts)` → 마커용 lean payload 일괄. 핀 표시·필터·정렬·like_count 까지 이걸로 다 처리.
- **Tier 2 — 핀 클릭 시** · `[fetchCafeDetail(id)](../lib/api/cafes.ts)` → 그 카페 *한 건* 만 detail view 에서. React Query 가 cafe별 캐시 키로 들고 있다가 같은 핀 재클릭 시 재사용.

### 결과

- 첫 페인트 페이로드가 **카페 수와 무관하게 마커 7컬럼으로 고정**. 카페 50개나 500개나 초기 페이로드 차이가 선형이지만 *상수가 훨씬 작다*.
- 클릭 안 한 카페의 detail 은 영영 fetch 안 됨 — 대부분의 카페는 평생 안 클릭됨을 활용.
- React Query 캐시도 가벼움. 마커 array 1개 + 클릭한 카페별 detail entries.

### 배운 점

- **데이터 모델을 "한 테이블 = 한 응답" 으로 고정시킬 필요 없다** — 사용 사례별로 view 를 만드는 게 훨씬 깔끔하다. 어차피 Postgres view 는 거의 무료고, ORM 처럼 객체 단위로 끌어올 이유도 없다.
- **모든 카드/리스트 컴포넌트에서 like_count 가 *반드시* 마커 페이로드에 있어야 한다는 제약**을 처음엔 놓쳤다. 좋아요 정렬·UI 표시 때문에. detail 만 갖고 있으면 마커 단계에서 정렬을 못 한다. 그래서 `like_count` 는 마커 view 의 7컬럼에 포함시켰고, 이게 #1 (비로그인 좋아요) 의 "마커 캐시 부분 패치" 와 자연스럽게 맞물렸다.
- 두 Tier 의 캐시 키를 분리한 게 결국 #1 의 부분 패치 전략이 가능했던 전제다. **여러 의사결정이 서로 강하게 맞물려 있을 때, 처음부터 정답이 보이진 않는다 — 한 곳에서 비용을 치르면 다른 곳에서 받아낸다는 감각이 중요했다.**

## 추가로 기록해두고 싶은 것

- **인증 정책의 비대칭** — "보는 건 자유, 참여도 일부 자유" 라는 슬로건은 깔끔하지만, 실제 구현하면서 "*어디까지 자유*" 의 경계를 그을 때마다 미세하게 흔들렸다. 후기는 비로그인 OK, 즐겨찾기는 로그인 필요, 좋아요는 비로그인 OK + localStorage, 사진 제보는 비로그인 OK. 각 결정의 근거를 의식적으로 적어두지 않으면 일관성이 깨진다.
- **Naver 지도 SDK 의 React 통합** — 지도는 React 트리 밖에 사는 외부 인스턴스라서, "지도가 마운트되기 전에 마커를 그리려는 경합" 이 종종 발생. `useNaverMap` 훅에서 지도 인스턴스 ready 상태를 명시적인 boolean 으로 노출해 의존 훅들이 기다리게 만들었다. SDK 통합은 "ready signal 을 hook return 에 포함시켜라" 가 정답.
